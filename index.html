Security Research

I built DataForest Learning as a deliberately vulnerable learning platform to demonstrate common web security anti-patterns. Below is my audit documenting every weakness — from exposed credentials to missing access controls — along with a full data scraping proof-of-concept.

CRITICAL

1. Hardcoded Admin Password in Client Bundle

Where in the code: The app initialises a React SPA with a single <div id="root">. All logic is in the bundled JS. The function dbAdminPw fetches the password from Supabase, but provides a hardcoded fallback if the query fails:

const dbAdminPw = async () => {
  const { data } = await SB.from("admins")
    .select("password").limit(1).single();
  return data?.password || "admin123";  // hardcoded fallback
};

How to find it: Open DevTools → Sources → search for "admin" in the bundled JS. The string "admin123" appears as a literal.

Why this is critical: Even if the database were unreachable or RLS blocked the query, the fallback guarantees the password is always readable in the client bundle. The password never needs to be guessed — it's shipped to every visitor.

Impact: Full admin access to any visitor who opens DevTools
Fix: Server-side auth with hashed passwords (e.g. Supabase Auth, bcrypt). Never embed secrets in client code.
CRITICAL

2. Exposed Supabase Anon Key — No Row-Level Security

Where in the code: The Supabase client is initialised at the top of the bundle with both the project URL and anonymous API key as string literals:

const SB = createClient(
  "https://cvngrkb...supabase.co",   // project URL
  "eyJhbGciOiJIUzI1NiIs..."          // anon key — visible to everyone
);

How to exploit: Copy the URL and key from the bundle, then query any table directly. No authentication required:

curl 'https://<project>.supabase.co/rest/v1/admins?select=*' \
  -H 'apikey: <anon_key>'

# Response: [{"id":1,"password":"admin123"}]

curl -X DELETE '.../rest/v1/projects?id=eq.<any_id>' \
  -H 'apikey: <anon_key>'

# Response: 204 No Content — row deleted, no auth needed

Why this is critical: Supabase anon keys are designed to be public — but only when RLS policies restrict what they can access. Here, no RLS policies exist on any table. The anon key grants unrestricted SELECT, INSERT, UPDATE, and DELETE on admins, users, projects, and the cores-images storage bucket.

Confirmed operations: SELECT (200), UPSERT (201), DELETE (204) — all succeed with the anon key
Fix: Enable RLS on every table. Write policies that scope reads/writes to authenticated roles. The anon key should only grant access the public is meant to have.
CRITICAL

3. Client-Side Authentication — No Server Validation

Where in the code: The admin login flow is a two-step client-side process. First, the password is fetched from the DB (or falls back to the hardcode). Then the comparison happens entirely in the browser:

// Step 1: Password is stored in React state
const [adminPw, setAdminPw] = useState(null);
useEffect(() => { dbAdminPw().then(setAdminPw); }, []);

// Step 2: Comparison is a local string check
const tryAdmin = () =>
  pw === adminPw
    ? onAdmin()           // sets screen state to "admin"
    : setPwErr("Incorrect password.");

// onAdmin just flips a React state variable:
const onAdmin = () => setScreen("admin");

How to bypass — 3 independent methods:

// Method 1: Read adminPw from React component state in DevTools

// Method 2: Call the screen setter directly from the console
// Find the React fiber, or simply re-render with screen="admin"

// Method 3: Skip the UI entirely — use the anon key to call
// Supabase REST API for any admin operation (CRUD on all tables)

Why this is critical: There is no session token, no JWT, no server-side middleware. The "admin" state is just a React variable. Every API call the admin panel makes uses the same anon key that every visitor already has. Authentication is cosmetic — it hides UI elements but protects nothing.

Impact: Auth bypass via console, DevTools, or direct API calls
Fix: Use Supabase Auth with email/password sign-in. Issue JWTs on login. Validate the JWT on every API call via RLS policies tied to auth.uid().
HIGH

Stored XSS via Rich Text Editor

Slide content from the Quill editor is rendered with dangerouslySetInnerHTML. An admin (or anyone, given the auth bypass) can inject arbitrary HTML/JS that executes for every learner who opens the slide.

Impact: Session hijacking, phishing overlays, keylogging
Fix: Sanitize HTML with DOMPurify before rendering; enforce a CSP
HIGH

Plaintext Password Storage

The admins table stores the password as a raw string ("admin123") — no hashing, no salting. Combined with the lack of RLS, the plaintext password is directly queryable by anyone.

Fix: Hash with bcrypt/argon2; never store or transmit plaintext credentials
HIGH

Full Database Write Access for Anonymous Users

The anon key permits INSERT, UPDATE, and DELETE on all tables. An attacker can wipe all courses, tamper with learner progress, or inject malicious course content — all without authentication.

# DELETE returns 204 — no auth required
curl -X DELETE '.../rest/v1/projects?id=eq.<any_id>' \
  -H 'apikey: <anon_key>'   # 204 No Content ✓
MEDIUM

No Content Security Policy

The app serves no Content-Security-Policy header. Combined with the stored XSS vector, there is nothing preventing injected scripts from exfiltrating data to external domains.

MEDIUM

No Input Validation or Sanitization

Learner names, media URLs, and quiz feedback are accepted as-is with no length limits, format checks, or encoding. This widens the attack surface for injection and abuse.

LOW

No Rate Limiting or Brute-Force Protection

The admin login has no lockout, CAPTCHA, or rate limit. Even if the password weren't already exposed, it could be brute-forced trivially.

Attack Chain Summary

1 Open DevTools → read anon key + Supabase URL from bundle
2 Query admins table directly → get plaintext password
3 Log in as admin, or skip login entirely by calling onAdmin()
4 Full control: edit/delete courses, exfil user data, inject XSS payloads
POC

Data Scraping Proof-of-Concept

To demonstrate the real-world impact of the missing access controls, I scraped the entire "Working With Data Platforms" course — all metadata, chapters, slides, body content, quiz questions, correct answers, and feedback — using a single unauthenticated API call.

Method: One curl request to the Supabase REST API with the publicly exposed anon key. No login, no session, no rate limiting encountered.

curl 'https://<project>.supabase.co/rest/v1/projects?select=*' \
  -H 'apikey: <anon_key>' | python3 parse_to_markdown.py

# Result: 12,424 bytes — the entire course in structured Markdown

What was extracted:

2 Chapters
8 Slides
12K Bytes of content
1 Quiz with answers

Extracted content includes:

  • Full course metadata (title, subtitle, duration, level, description)
  • Chapter structure with tags and durations
  • All slide headings and full body text (HTML converted to Markdown)
  • All 3 real client case studies (RMC-Norkem, NexGen Cloud, Sagis Diagnostics)
  • Quiz questions, all answer options, correct answer flags, and per-option feedback
  • All learner names, progress data, and quiz responses from the users table

The complete scraped output is available as scraped_course_data.md in this repository.

Security features present: None. No authentication, no rate limiting, no API scoping, no content obfuscation.
Security features absent: RLS policies, API key scoping, authentication middleware, CORS restrictions, rate limiting, content encryption, access logging.
Security features insufficient: The anon key exists (Supabase provides it) but is paired with zero RLS policies, making it equivalent to a public read/write API.
